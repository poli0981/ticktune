# 08 — Internationalization (EN/VI)

Suite 1.0 · 2026-07-21

Vietnamese is the default; English is first-class. Two mechanisms, matched to page
type:

## 1. Static pages — route-based

- `/` = VI, `/en/` = EN mirrors (landing, `legal/*`, 404), following the
  `poli0981.dev` pattern.
- Astro content: one component per page, strings from
  `src/i18n/static/{vi,en}.ts` typed dictionaries; `hreflang` alternates +
  `lang` attribute per route.
- Legal pages: **English is canonical** for interpretation (stated on the VI
  pages); both languages ship at v1.0 — VI translation of legal drafts is a
  roadmap P6 task (`16 §P6`).

## 2. App island — runtime i18next

- `i18next@26`, resources bundled from `src/app/i18n/{en,vi}.json` (no HTTP
  backend, **no detector plugin** — a one-time `navigator.language` sniff at
  init, described below, and an explicit toggle after that).
- Initial language: persisted setting → else `navigator.language` starts with
  `vi` → VI, else EN. Implemented once, in `initialLang`
  (`tt-settings-schema.ts`), and the runtime is *told* the result rather than
  deciding it again — two mechanisms answering the same question is how they
  come to disagree.
- Toggle in Z6 header; switch is instant (no reload); persisted to Dexie;
  `document.documentElement.lang` updated (a11y, `03 §8`).

### 2.1 Runtime settings, stated rather than left to the implementer

Filed 2026-07-23, closing the 🟡 `AUDIT-BACKLOG` finding that named these as
undefined. Both are load-bearing, not defaults worth inheriting:

| Setting | Value | Why |
|---------|-------|-----|
| `interpolation.escapeValue` | **`false`** | `09 §4` bans `{@html}` and Svelte already escapes every interpolation. i18next's default `true` would escape a second time, so a track called *Bird & Boy* renders as `Bird &amp;amp; Boy` — and every interpolated value here is a **track title**, which is user-supplied text |
| `fallbackLng` | **`'en'`** | Not `vi`, even though VI is the UI default. The fallback is the **reference** dictionary: `TtKey` is derived from `en.json`, so EN is the only file guaranteed to hold every key. A key missing from VI renders its English text rather than a raw id |
| missing key | renders the **EN value**, never the id | The key-diff guard (`§3`, `13 §1`) is build-failing, so this should be unreachable; it defines what it looks like if it ever is. `returnNull: false` keeps a null value from reaching the DOM as the string `"null"` |

⚠️ **`escapeValue: false` is not a style choice and must not be "hardened" back.**
Turning it on does not add safety — Svelte's escaping is what provides that — it
only corrupts non-ASCII and punctuation in exactly the strings this app is built
to display.

## 3. Key conventions

- Namespace by feature: `setup.*`, `player.*`, `settings.*`, `yt.err.*`,
  `gate.*`, `toast.*`, `finished.*`. The track-info modal (`02 §8`) files under
  **`player.trackinfo.*`** rather than a namespace of its own.
- Keys are stable IDs (`yt.err.blocked.title`), never English sentences.
- Interpolation via i18next `{{count}}`/plurals — VI has no plural forms
  (single form), EN uses `_one/_other`.
- Numbers/dates/durations: `Intl.NumberFormat`/`Intl.DateTimeFormat` with the
  active locale; countdown digits are locale-independent by design.
- `N/A` and `–` fallbacks are **not** translated (universal per portfolio
  convention).
- CI guard: a Vitest test diffs key sets of `en.json` vs `vi.json` — any
  missing/extra key fails the build (`13 §3`).

### 3.1 Keys filed ahead of P5 — ✅ **wired 2026-07-23**

The table below is what P2–P4 filed while every string was a literal. P5 built
the dictionaries from the **shipped strings**, so it is a superset of this list:
`gate.*` (the legal gate was never filed), the whole of `player.queue.*` and
`player.menu.*`, the Setup offline banner and its five start-hint variants,
`finished.unit.*`, and three trackinfo **values** — `sourceLocal`,
`sourceYoutube`, `coverArtYes` — that "one key per `02 §8` field" did not cover
because they are values rather than labels.

⚠️ Two `Intl` call sites hardcoded `'vi-VN'` until P5 wired `i18n.locale`, so an
EN user got Vietnamese dates. `§3`'s "with the active locale" was the rule the
whole time; nothing enforced it.

The **statics** (`§1`) are untouched: `/`, `/404` and `TtBase.astro` are still
Vietnamese literals, because route-based EN mirrors are P6's landing work.

#### `settings.*` — added by P5 slice 2

`§3`'s namespace list has named `settings.*` since suite 1.0 and it was empty
until the panel existed. It now covers the seven groups `03 §6` ships (`group.*`,
`general.*`, `countdown.*`, `audio.*`, `playback.*`, `hotkeys.*`,
`diagnostics.*`, `about.*`), plus `header.settings` / `header.fullscreen` for the
two new Z6 buttons and `player.focus.*` for the 3 s hint chips.

⚠️ **The orphan guard shaped the markup, and that is working as intended.** It
finds callers by grepping for a literal `t('…')`, so three things that would
naturally have been loops are written out instead: the eleven-row hotkey table,
the `endAction` options, and the log-level filter. The hint chip likewise renders
through an `{#if}` on a `'focus' | 'railLocked'` tag rather than storing a
`TtKey`, because a key held in a variable has no greppable caller. **No new
`CALLED_INDIRECTLY_PREFIX` entry was added**, and none was needed.

Two values stay untranslated for `§3`'s reason: the SPDX id `GPL-3.0-only`, which
is an identifier rather than prose, and the language buttons' own labels
(`Tiếng Việt` / `English`), each written in the language it selects.

`04 §2` item 5 authorises P2 to hardcode Vietnamese and file its keys rather than
stand up i18next a phase early (which knip would fail anyway — an unused
dictionary is an unused file). P2's strings are therefore literals in components,
and these are the keys P5 must create when it wires the runtime dictionary:

| Namespace | Keys |
|-----------|------|
| `finished.*` | `title`, `late.title`, `late.at`, `late.ago`, `summary.tracks`, `summary.duration`, `action.restart`, `action.back` |
| `setup.*` | `mode.single/playlist/youtube`, `mode.locked.hint`, `countdown.hours/minutes/seconds`, `countdown.presets`, `countdown.match`, `drop.idle/active/reject`, `start` |
| `player.*` | `loop.count`, `loop.style.hard/crossfade`, `loop.style.locked`, `transport.play/pause/prev/next/mute`, `position` |
| `player.trackinfo.*` | one key per `02 §8` field + `open`, `close` |
| `toast.import.*` | `summary`, `code.TT-IMP-001` … `TT-IMP-008`, `code.TT-PLY-100`, `code.TT-PLY-101`, `code.TT-PLY-103` |

`N/A` and `–` stay untranslated per `§3`, and the countdown digits are
locale-independent by design — so nothing in `04 §4` needs a key.

## 4. Tooling continuity

Dictionaries are plain i18next JSON — directly compatible with the existing
JSONPrism i18next→RESX bridge if strings are ever shared with desktop projects.

## 5. Adding a language later

Post-1.0 JP would follow the portfolio pattern: add `ja.json` + `/ja/` static
routes; no code changes beyond the language menu. Not scheduled for v1.0.
